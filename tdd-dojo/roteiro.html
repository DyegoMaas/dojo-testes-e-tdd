PASSO 01 / 03

🔴 RED

Escreva UM teste que falha. Só um. Rode e veja-o falhar — falhar pelo motivo certo.

[Fact]
public void Add_DoisNumeros_RetornaSoma()
{
    calculator.Add("1,2").ShouldBe(3);
}
// dotnet test → FAILED ✗
// perfeito: é exatamente isso que queremos

PASSO 02 / 03

🟢 GREEN

Faça passar com o MÍNIMO de código. Pode ser feio. Pode ser hardcoded. Sério.

public int Add(string numbers)
{
    return 3; // hardcode é PERMITIDO aqui 😈
}
// dotnet test → PASSED ✓
// o próximo teste vai forçar a generalização

PASSO 03 / 03

♻️ REFACTOR

Limpe o código com os testes verdes te protegendo. Nomes, duplicação, intenção. Rode os testes de novo.

public int Add(string numbers) =>
    numbers.Split(',').Sum(int.Parse);

// dotnet test → PASSED ✓
// continua verde? então o refactor é seguro

As 3 Regras
de Ouro

REGRA 01

Nenhuma linha sem teste vermelho

Nenhuma linha de produção sem um teste vermelho pedindo por ela. Se você se pegou escrevendo código "que vai precisar" — apague.

REGRA 03

Um teste por vez

Nada de escrever cinco testes e sair implementando. Um vermelho, um verde, respira, repete.

★ conceito novo

REGRA 02

👣 Baby Steps — passos de bebê

O menor passo que te leva de volta ao verde. Hardcode é permitido — e até incentivado — no GREEN. A generalização vem do próximo teste, e jamais da sua imaginação. Você não escreve o algoritmo: você deixa os testes arrancarem o algoritmo de você, um caso por vez.

🔴 Add("") → 0 🟢 return 0; 🔴 Add("1") → 1 🟢 if vazio 0, senão Parse ♻️ agora sim, generaliza

"Mas isso é burrice, eu já sei a solução!" — Saber a solução não é o ponto. O ponto é treinar o músculo de avançar em passos tão pequenos que você nunca fica perdido.

Kata 1 · Randori · 60 min

🥋 String Calculator

int Add(string numbers)
produção → src/StringCalculator/Calculator.cs testes   → tests/StringCalculator.Tests/CalculatorTests.cs (o 1º teste já está lá, vermelho)
0/8 requisitos

Revele um requisito por vez. Não espie o próximo antes de fechar o atual — a graça do dojo é o requisito te surpreender.

  • Requisito 1 — string vazia já no repo
    String vazia retorna 0. → Add("") ➜ 0. Já implementado como exemplo: use-o para demonstrar o ciclo.
  • Requisito 2
    Um único número retorna ele mesmo. → Add("1") ➜ 1, Add("42") ➜ 42
    🔒 revelar requisito
  • Requisito 3
    Dois números separados por vírgula são somados. → Add("1,2") ➜ 3
    🔒 revelar requisito
  • Requisito 4
    Uma quantidade qualquer de números. → Add("1,2,3,4,5") ➜ 15
    🔒 revelar requisito
  • Requisito 5
    Quebra de linha também separa números (junto da vírgula). → Add("1\n2,3") ➜ 6
    🔒 revelar requisito
  • Requisito 6
    Delimitador customizado no início, no formato //[delim]\n[números]. → Add("//;\n1;2") ➜ 3
    🔒 revelar requisito
  • Requisito 7
    Números negativos são proibidos: lance uma exceção cuja mensagem liste os negativos encontrados. → Add("1,-2,-5") ➜ lança, mensagem contém -2 e -5. 
    var ex = Should.Throw<ArgumentException>(() => calculator.Add("1,-2,-5"));
ex.Message.ShouldContain("-2");
ex.Message.ShouldContain("-5");
    🔒 revelar requisito
  • Requisito 8 extra
    Números maiores que 1000 são ignorados. → Add("2,1001") ➜ 2
    🔒 revelar requisito
💡 Para um time muito novato, parar no Requisito 5 ou 6 já é um dojo completo e vitorioso. Melhor sair com o ciclo dominado do que correr até o 8.

Kata 2 · Ping-pong · 60 min · temática IA

🥋 Extrator de JSON

Result<string> Extract(string raw)
produção → src/JsonHarness/JsonExtractor.cs testes   → tests/JsonHarness.Tests/JsonExtractorTests.cs (1º teste já vermelho) apoio    → src/JsonHarness/Result.cs (pronto, não precisa mexer)
0/6 requisitos

A entrada é o texto cru que um LLM devolve; a saída é o JSON válido isolado, ou uma falha graciosa. É função pura — sem rede, sem LLM, sem NSubstitute. Cada teste é um jeito diferente de um modelo baratinho se comportar mal.

  • Requisito 1 — JSON limpo já no repo
    JSON limpo entra → sucesso com o mesmo conteúdo. → Extract("{\"nome\":\"Ada\"}") ➜ Success("{\"nome\":\"Ada\"}")
  • Requisito 2
    O modelo embrulhou em cerca de markdown. Extraia só o JSON de dentro. 
    ```json
{"nome":"Ada"}
```
    ➜ Success("{\"nome\":\"Ada\"}")
    🔒 revelar requisito
  • Requisito 3
    O modelo tagarelou antes e depois. Isole o objeto. → Extract("Claro! Aqui está: {\"nome\":\"Ada\"} 😊") ➜ Success("{\"nome\":\"Ada\"}")

    (Dica que o teste vai forçar: localizar o primeiro { e o último } e validar o miolo. Mas só implemente quando o teste pedir.)
    🔒 revelar requisito
  • Requisito 4
    Lixo ou string vazia → falha graciosa, sem lançar exceção. → Extract(""), Extract("desculpe, não consegui") ➜ IsFailure == true
    🔒 revelar requisito
  • Requisito 5
    JSON truncado/inválido (o modelo cortou no meio) → falha graciosa. → Extract("{\"nome\":\"Ada") ➜ IsFailure == true

    (Aqui entra a validação de verdade: JsonDocument.Parse + tratar o JsonException. Isso separa "achei chaves" de "é JSON válido".)
    🔒 revelar requisito
  • Requisito 6 extra
    O payload é um array no topo, e nada de objeto. → Extract("[1,2,3]") ➜ Success("[1,2,3]"). Force a generalização de "procurar {...}" para "procurar o primeiro valor JSON válido".
    🔒 revelar requisito
🌉 Ponte para a Sessão 2: quando o grupo perceber que Extract é fácil de testar porque não fala com o LLM, você tem o gancho perfeito. Na próxima sessão, o passo que chama o modelo (lento, caro, não-determinístico) é escondido atrás de uma interface e mockado com NSubstitute. Aí DI e mock deixam de ser cerimônia e viram alívio.
